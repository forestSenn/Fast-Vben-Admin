## 2. 背景与结论

当前项目已经具备完整的多租户基础骨架：

- `Tenant` 负责租户主实体与生命周期骨架。
- `TenantPlan` 负责配额与资源上限。
- `TenantInitializationTemplate` 负责租户初始化策略。
- `TenantProfile`、`TenantPlanProfile`、`TenantPlanMenu` 已经为扩展运营字段、套餐画像、套餐菜单授权预留了模型空间。

参考项目 `LZ-litchi` / `LZ-litchi-ui-admin-vben` 的价值主要体现在两点：

- 租户管理页更偏运营视角，字段更完整，包含联系人、手机号、行业、地址、资质、域名、账号数、创建管理员等信息。
- 套餐管理并不只是“配额”，而是带有菜单授权能力的“权限套餐”，并提供套餐授权、租户菜单更新、全量租户菜单同步等动作。

同时也需要明确一个关键事实：

- 参考项目没有与当前仓库完全等价、独立可运营的“初始化模板管理”模块。
- 参考项目的租户运营能力不是全部堆在租户主表，而是拆成“租户资料、套餐权限、套餐订阅”三层。
- 其中试用/服务周期、订阅状态、支付状态、价格与折扣等交易属性主要由套餐订阅层承载。
- 租户响应中的充值金额、支付金额、余额金额属于汇总展示字段，不是租户创建/编辑字段。
- 当前项目的初始化模板是已有差异化能力，不应因为参考项目没有同形态实现而被削弱或移除。

但结合当前项目目标，本次专题的第一优先级并不是“把套餐做得更像参考项目”，而是让“租户管理真正具备运营抓手”。套餐授权增强是配套能力，不应喧宾夺主。

本 PRD 的核心结论是：

- 不重做当前租户域架构，不把 `TenantPlan` 改造成参考项目那种单一“权限套餐”模型。
- 保留当前三段式职责分工：`租户`、`套餐/配额`、`初始化模板`。
- 以“租户生命周期运营”为主轴，吸收参考项目的高价值能力：租户运营字段、套餐菜单授权、菜单同步动作、后台页面操作完整度。
- 把“初始化模板”定位为平台高级配置，不再与“套餐授权”争夺主叙事。

### 2.1 参考项目真实分层

| 参考项目对象 | 核心职责 | 对当前项目的启发 |
| --- | --- | --- |
| 租户 | 名称、编码、联系人、行业、类型、地址、资质、网站、账号数、管理员开通信息 | 补齐租户运营资料与开户体验 |
| 租户套餐 | 套餐画像、发布状态、价格、菜单授权 | 在现有配额套餐上增加菜单基线，不删除配额能力 |
| 套餐订阅 | 订阅状态、支付状态、开始时间、结束时间、天数、价格、折扣价、总价 | 真正形成续费、订单、支付历史后，应独立建模 |
| 角色菜单 | 用户最终功能权限 | 套餐菜单同步最终仍落到角色菜单，不建立第二套最终权限源 |

因此，“完全参考”应理解为复用它的职责拆分和运营动作，而不是把所有参考字段复制到当前租户表。

## 3. 目标与非目标

### 3.1 目标

本次增强要达成以下目标：

1. 让租户管理从“基础数据维护”升级为“可运营、可开通、可审计”的后台能力。
2. 让租户管理具备生命周期字段和运营动作，包括试用、正式、冻结、到期、续期、负责人、来源与跟进备注。
3. 让套餐管理在保留配额能力的同时，补齐菜单授权能力，形成“配额 + 功能权限”的混合套餐模型。
4. 让租户与套餐之间具备显式的菜单同步链路，避免套餐变更后租户权限长期漂移。
5. 保留初始化模板能力，并把它明确为租户开通时的初始化策略中心。
6. 在不破坏现有 API、表结构语义和前端路由结构的前提下完成能力增强。

### 3.2 非目标

本次不进入范围的内容：

- 不重构为参考项目同构的 Java/Spring 风格租户体系。
- 不取消或弱化初始化模板模块。
- 不在一期内引入完整订阅订单、支付流水、账单中心。
- 不在一期内实现域名自动绑定、ICP备案校验、短信/邮件开户自动化。
- 不在一期内做复杂工作流审批。
- 不强行改动现有多租户隔离主链路和上下文切换机制。

| 前端页面 | 已具备基础 | 已有租户、套餐、模板三个页面，但交互仍偏轻量 |

| 问题 | 影响 |
| --- | --- |
| 租户页字段偏少，运营信息不完整 | 无法支持商务开通、售后排查、运营检索 |

| 前端操作链路偏弱 | 缺少菜单授权抽屉、同步动作、创建租户时的开通体验 |
| 字段虽然在模型层有预留，但 API/页面未完全打通 | 造成“看起来有扩展点，但不可用”的落差 |

## 5. 产品原则

### 5.1 保留现有架构

三类对象职责必须保持清晰：

- `Tenant`：谁是租户、租户是否启用、租户当前绑定哪个套餐和模板。
- `TenantPlan`：这个租户能用多少资源、默认具备哪些系统功能菜单。
- `TenantInitializationTemplate`：创建租户时要初始化哪些组织与基础配置。

### 5.2 吸收参考项目优点，而不是复制实现

可以借鉴参考项目的字段、操作入口和页面组织，但不照搬其后端对象语义。当前项目需要的是“增强型混合套餐”，不是把 `TenantPlan` 退化成纯菜单包。

### 5.3 运营闭环优先于字段堆砌

租户运营化不是简单增加几个资料字段，而是要让运营人员能回答以下问题：

- 哪些租户正在试用，何时到期。
- 哪些租户即将续费或已经过期。
- 哪些租户被冻结，原因是什么，谁负责跟进。
- 当前租户属于谁在跟，来自什么渠道，绑定了什么套餐。

### 5.4 角色权限仍是最终权限源

一期不新建“租户最终权限”第二套来源。用户实际权限仍来自租户内角色的 `RoleMenu`。

### 5.5 套餐授权优先于人工散改

租户菜单能力的主来源应当是套餐授权。人工对单租户做菜单例外调整时，必须保留可追踪性，并能重新同步回套餐基线。

### 5.6 模板负责初始化，不负责长期授权

初始化模板只负责“创建时生成什么”，不承担长期套餐权限管理职责。

### 5.7 一期简化不锁死长期模型

一期允许在租户运营扩展中保存“当前试用截止时间、当前服务到期时间”等快照字段，用于筛选和运营动作；但这些字段不承担完整订阅历史。

出现以下任一需求时，应引入独立的 `TenantSubscription`（或 `TenantLease`）领域对象：

- 一个租户需要保留多次续费、升级、降级记录。
- 需要区分订阅状态、支付状态、订单状态。
- 需要记录原价、折扣价、实付金额、退款金额。
- 套餐变更需要预约生效或按时间段生效。
- 财务对账、发票、账单开始依赖租户服务周期。

届时租户页展示“当前有效订阅”的汇总结果，订阅记录负责历史和交易事实，避免租户资料表成为事实流水表。

## 6. 目标用户与核心场景

| 角色 | 诉求 | 典型场景 |
| --- | --- | --- |
| 平台超级管理员 | 统一开通租户、配置套餐、维护初始化模板 | 新签客户开通、套餐升级、排查权限问题 |
| 运营人员 | 查询租户经营信息和当前能力 | 查看联系人、账号数、套餐能力、是否启用 |
| 销售/客户成功 | 跟进试用与续费状态 | 查看负责人、到期时间、来源渠道、跟进备注 |
| 实施/交付人员 | 快速创建客户租户并初始化默认数据 | 创建租户时同步生成管理员和组织骨架 |
| 技术支持 | 定位租户为何看不到某菜单、为何超配额 | 排查套餐菜单授权、配额上限、模板初始化差异 |

核心业务场景：

1. 超级管理员创建租户时，填写基础信息、生命周期信息、联系人、行业、账号数、管理员账号，选择套餐与初始化模板后完成开通。
2. 运营人员可以筛出试用中、即将到期、已过期、已冻结的租户，并查看负责人和跟进信息。
3. 超级管理员维护套餐时，既能管理配额，又能配置该套餐默认可访问菜单。
4. 套餐菜单变更后，管理员可以选择同步当前套餐关联租户内置角色菜单，或执行全量同步。
5. 初始化模板用于定义租户开通时的默认组织和系统配置，不直接承担套餐授权职责。

## 7. 功能需求

### 7.1 租户管理增强

目标：让租户管理页具备参考项目的运营化信息密度，同时兼容当前架构。

#### 7.1.1 生命周期运营

租户管理一期必须支持以下生命周期字段：

- 租户状态：试用中、正式、已冻结、已过期、已归档。
- 生效时间。
- 试用截止时间。
- 服务到期时间。
- 冻结原因。
- 负责人。
- 客户来源。
- 跟进备注。

一期需要支持的生命周期动作：

- 开通试用租户。
- 试用转正式。
- 手动续期。
- 冻结租户。
- 解冻租户。
- 归档租户。
- 更换负责人。

说明：

- 生命周期状态必须可搜索、可筛选、可审计。
- 到期和冻结属于运营状态，不应仅依赖 `is_active` 单布尔值表达。
- 一期续期是对“当前服务周期快照”的轻量调整，不等同于完整的订阅订单流程。
- 一期必须记录续期、冻结、解冻、转正式等操作日志，弥补尚未引入订阅历史表的限制。

#### 7.1.2 字段增强

租户管理页应支持以下信息展示与维护：

- 基础字段：租户名称、租户编码、状态、描述、创建时间。
- 关联关系：当前套餐、初始化模板。
- 联系信息：联系人、联系电话、联系用户。
- 经营属性：行业、租户类型、账号数量。
- 地址资质：地区编码、详细地址、资质说明。
- 外部信息：绑定域名/网站。
- 使用情况：当前成员数。
- 开通信息：创建管理员账号、初始密码。
- 运营信息：负责人、客户来源、跟进备注。
- 账务概览：充值金额、支付金额、余额金额。

说明：

- 创建管理员账号与初始密码仅在“创建租户”时必填或按策略填充，编辑时不再回显明文密码。
- 账务概览字段一期只允许作为只读展示占位，来源应为后续独立账务模块或汇总流水，不允许在租户资料页手工维护。

#### 7.1.3 列表与筛选

列表页至少支持：

- 关键字搜索：租户名称、租户编码、联系人、手机号。
- 生命周期状态筛选。
- 套餐筛选。
- 初始化模板筛选。
- 行业筛选。
- 负责人筛选。
- 来源筛选。
- 到期时间筛选。
- 即将到期快捷筛选，例如 7 天内、30 天内。

#### 7.1.4 操作能力

租户管理页一期需支持：

- 创建租户。
- 编辑租户。
- 试用转正式。
- 续期。
- 冻结/解冻。
- 更换负责人。
- 归档租户。
- 查看租户当前配额与使用量。
- 查看租户当前套餐菜单基线。
- 手动触发“按当前套餐同步租户内置角色菜单”。

二期预留：

- 单租户自定义菜单覆盖。
- 重置租户管理员密码。
- 变更租户默认模板并重建部分基础数据。

### 7.2 套餐管理增强

目标：把当前“配额型套餐”升级为“配额 + 菜单授权”的混合套餐。

#### 7.2.1 套餐基础字段

套餐保留现有字段并补齐扩展画像：

- 现有字段：套餐名称、套餐编码、成员上限、文件数上限、存储上限、描述、默认套餐、启用状态。
- 扩展字段：套餐类型、logo、价格、发布状态、排序号、订阅数、订阅总额、备注。

说明：

- 配额字段继续保留，不能因对齐参考项目而删除。
- 扩展字段优先作为运营视图和页面展示能力，不要求一期全部接入业务计算链路。

#### 7.2.2 菜单授权

套餐管理必须新增“菜单授权”能力：

- 可为套餐配置菜单树授权。
- 可读取套餐当前已授权菜单。
- 保存套餐菜单授权后持久化到 `TenantPlanMenu`。
- 默认套餐也允许授权，但不允许取消所有基础必需菜单。

授权原则：

- 套餐菜单是租户内置角色的默认功能基线。
- 菜单授权应复用现有系统菜单树与 RBAC 菜单表，不另起一套权限源。
- 一期用户最终权限仍通过租户内角色的 `RoleMenu` 生效。
- 一期同步动作的落点是更新租户内置角色，例如 `super_admin`、`admin`、`user` 的菜单基线，而不是新建第二套最终权限关系。
- 自定义角色一期不自动覆盖，避免误伤租户自定义授权。
- 按钮权限是否纳入套餐授权，一期采用“跟随菜单树可选授权”的方案，由技术设计决定最终颗粒度。

#### 7.2.3 同步动作

套餐管理页需要新增以下操作：

- 同步当前套餐关联租户内置角色菜单。
- 同步时输出结果摘要：成功数、失败数、跳过数。

说明：

- 若一期不实现异步任务中心，至少需要同步结果反馈。
- 同步动作默认只影响“跟随套餐基线”的租户，不强制覆盖未来可能存在的人工例外配置。

### 7.3 初始化模板定位收敛

目标：保留模板能力，同时避免它与套餐能力职责重叠。

#### 7.3.1 模板职责

初始化模板继续负责以下内容：

- 根部门编码和名称。
- 是否初始化岗位。
- 是否初始化字典。
- 是否初始化系统参数。
- 是否初始化存储渠道。
- 是否初始化消息模板。
- 是否初始化短信渠道。
- 是否初始化邮件账户。

#### 7.3.2 模板展示策略

页面与文案需要体现：

- 模板用于“开通时初始化基础环境”。
- 模板不是套餐，不承担长期菜单授权和配额约束。
- 默认模板必须存在且不可随意删除。

#### 7.3.3 后续扩展方向

模板二期可考虑扩展：

- 默认角色种子包。
- 默认组织架构模板。
- 默认业务字典模板。
- 默认消息模板组合。

### 7.4 页面体验对齐

目标：在现有 Vben 页面结构下，吸收参考项目更完整的操作感。

需要补齐的页面体验：

- 租户表单分区展示：基础信息、生命周期信息、联系人信息、经营信息、初始化信息。
- 创建租户支持管理员账号信息输入。
- 套餐页增加“菜单授权”入口，采用抽屉或弹窗树授权。
- 套餐页列表增加运营字段展示，如价格、发布状态、订阅数。
- 租户页增加“续期”“冻结/解冻”“同步菜单”动作。
- 模板页加强“初始化用途”文案提示。

## 8. 数据与接口要求

### 8.1 数据模型要求

本次优先复用现有模型：

- `TenantProfile`：承载租户运营信息。
- `TenantPlanProfile`：承载套餐运营画像。
- `TenantPlanMenu`：承载套餐与菜单关系。

要求：

- 不新增语义重复表。
- 优先在现有表和 schema 上补齐 API 出参/入参。
- 对默认套餐、默认模板、系统级保底菜单建立后端保护。

一期生命周期字段建议落在 `TenantProfile` 或独立的一对一运营扩展中，不直接污染租户身份主字段。是否新建一对一扩展表，应在技术设计阶段根据现有 `TenantProfile` 字段密度和迁移成本决定。

中期演进边界：

- `Tenant` / `TenantProfile` 保存租户身份、联系方式、运营归属及当前状态快照。
- `TenantPlan` 保存配额与套餐菜单基线。
- `TenantSubscription` 保存套餐订阅周期、续费历史、价格和支付关联。
- `TenantInitializationTemplate` 仅保存开通初始化策略。

### 8.2 接口要求

一期建议补齐以下能力：

- 租户列表/详情返回完整运营字段。
- 创建/更新租户支持扩展字段写入。
- 创建/更新租户支持生命周期字段写入。
- 套餐列表/详情返回扩展画像字段。
- 套餐菜单查询接口。
- 套餐菜单保存接口。
- 基于套餐同步单租户内置角色菜单接口。

### 8.3 权限要求

建议权限码：

- `platform:tenant:query`
- `platform:tenant:create`
- `platform:tenant:update`
- `platform:tenant:delete`
- `platform:tenant:sync-menu`
- `platform:plan:query`
- `platform:plan:create`
- `platform:plan:update`
- `platform:plan:delete`
- `platform:plan:grant-menu`
- `platform:plan:sync-menu`
- `platform:template:query`
- `platform:template:create`
- `platform:template:update`
- `platform:template:delete`

## 9. 非功能要求

### 9.1 兼容性

- 保持现有租户创建、套餐创建、模板创建主流程可用。
- 老数据在不补录运营字段时也能正常读写。
- 不要求用户一次性补齐所有历史租户扩展信息。

### 9.2 审计性

- 套餐菜单授权变更应记录操作日志。
- 租户同步菜单动作应记录操作日志。
- 租户关键资料变更应保留审计记录。

### 9.3 可维护性

- 菜单授权逻辑与角色菜单授权尽量复用相同的菜单树与校验思路。
- 页面字段和接口字段命名尽量贴近现有模型，避免多层映射成本。

## 10. 验收标准

### 10.1 功能验收

1. 超级管理员可在租户管理页创建带联系人、行业、账号数、管理员账号的租户。
2. 运营人员可筛出试用中、已冻结、7 天内到期和 30 天内到期租户。
3. 租户列表可展示套餐名、模板名、联系人、手机号、账号数、负责人、到期时间、生命周期状态等关键字段。
4. 套餐管理可新增、编辑配额字段与运营字段。
5. 套餐管理可打开菜单授权界面并成功保存菜单树。
6. 套餐菜单变更后，可手动同步当前套餐租户内置角色菜单，并看到同步结果反馈。
7. 初始化模板仍可独立维护，且在创建租户时可正常选择并生效。

### 10.2 数据验收

1. 历史租户、套餐、模板数据升级后可正常查询。
2. 未补录扩展字段的数据不会导致页面报错。
3. 默认套餐与默认模板保护规则仍然有效。

### 10.3 权限验收

1. 无租户管理权限的用户不可访问租户管理接口。
2. 无套餐授权权限的用户不可执行菜单授权操作。
3. 无菜单同步权限的用户不可触发同步动作。

## 11. 成功指标

| 指标 | 目标 |
| --- | --- |
| 租户开通信息完整度 | 新建租户时可录入核心运营字段与生命周期字段 |
| 生命周期运营能力 | 可筛选试用、冻结、即将到期、已过期租户 |
| 负责人可追踪性 | 每个活跃租户都可明确负责人和来源 |
| 套餐授权能力 | 套餐可配置菜单树并成功持久化 |
| 菜单一致性 | 套餐变更后可通过同步动作让租户内置角色菜单回到基线 |
| 架构稳定性 | 不破坏现有 `Tenant` / `TenantPlan` / `TenantInitializationTemplate` 语义 |
| 页面可用性 | 系统管理内三页主流程均可独立完成操作 |

## 12. 产品决策摘要

本次方案不是“完全照搬参考项目”，而是“保留现架构、吸收参考项目优点”的专题增强：

- 租户管理吸收参考项目的运营字段和开通动作。
- 套餐管理吸收参考项目的菜单授权和同步动作。
- 初始化模板保留当前项目差异化价值，并明确其职责边界。
- 最终形成适合当前 FastAPI + Vben 架构的租户运营方案，而不是做一套语义冲突的拼装系统。
- 一期用生命周期快照快速形成运营闭环；当续费、支付和套餐变更需要保留业务历史时，再按参考项目思路引入独立订阅层。
