## 2. 开发目标

本计划用于指导“系统管理 > 租户管理 / 套餐管理 / 初始化模板”专题增强落地。目标不是推翻当前多租户设计，而是在现有架构上补齐产品化能力：

- 租户管理更运营化。
- 租户管理具备生命周期与负责人闭环。
- 套餐管理从纯配额升级到“配额 + 菜单授权”。
- 初始化模板职责更清晰。
- 套餐与租户之间新增菜单同步链路。
- 一期以租户生命周期快照形成运营闭环，同时为后续独立订阅层保留边界。

## 3. 实施原则

### 3.1 保留现有架构分层

- `Tenant` 继续负责租户主实体与生命周期。
- `TenantPlan` 继续负责配额主实体。
- `TenantInitializationTemplate` 继续负责初始化模板。
- 扩展信息优先落在 `TenantProfile`、`TenantPlanProfile`、`TenantPlanMenu`。

### 3.2 复用现有权限与菜单体系

- 套餐菜单授权复用现有 `Menu` 表与菜单树接口思路。
- 不创建第二套功能权限源。
- 用户最终权限仍通过 `RoleMenu` 生效。
- 一期的套餐同步落点是租户内置角色菜单基线，不直接覆盖自定义角色。

### 3.3 分阶段打通，不一次性做大而全

- 先打通数据与接口。
- 再补齐前端页面和交互。
- 最后做联动同步、测试、文档收口。

### 3.4 兼容存量数据

- 历史租户、套餐、模板数据必须平滑兼容。
- 扩展字段允许为空。
- 默认套餐、默认模板、系统保底菜单保持保护逻辑。

### 3.5 参考项目分层映射

参考项目实际采用“租户资料 + 套餐权限 + 套餐订阅 + 角色菜单”的协作方式。当前项目一期不直接复制完整订阅交易模块，但按以下边界实施：

| 当前对象 | 一期职责 | 不应承载的内容 |
| --- | --- | --- |
| `Tenant` / `TenantProfile` | 租户资料、负责人、来源、当前生命周期状态与时间快照 | 多次续费历史、订单、支付流水 |
| `TenantPlan` / `TenantPlanProfile` | 配额、套餐画像、价格展示、发布状态 | 单租户实际支付事实 |
| `TenantPlanMenu` | 套餐菜单基线 | 用户最终权限 |
| `RoleMenu` | 用户最终功能权限 | 套餐商业信息 |
| `TenantInitializationTemplate` | 开通时初始化策略 | 长期授权与续费状态 |

当出现多次续费、套餐升降级预约生效、支付/退款、账单或对账需求时，新增 `TenantSubscription`，不继续扩展租户资料表。

| 缺口 | 现象 |
| --- | --- |
| 租户扩展字段未完整打通 | 模型有字段，页面/API 未全面体现 |

| 菜单同步缺少管理动作 | 套餐变更不会主动作用到租户菜单 |
| 模板文案与定位不够清楚 | 容易与套餐职责混淆 |

| M1-BE-06 | 新增套餐菜单保存接口 | 同上 | 校验菜单存在性、去重保存、必要保底规则 | 保存成功后可立即重新读取 |
| M1-BE-07 | 新增单套餐同步接口 | 同上 | 将套餐菜单基线同步到关联租户内置角色菜单 | 返回同步统计结果且不覆盖自定义角色 |
| M1-BE-08 | 新增全量同步接口 | 同上 | 遍历全部租户或全部有效套餐关联租户 | 返回同步统计结果 |
| M1-BE-09 | 增加默认/系统保护 | 同上 | 默认套餐、默认模板、保底菜单不允许破坏性修改 | 关键保护规则测试通过 |

### 7.3 技术说明

建议采用以下实现策略：

- 租户查询统一通过 `build_tenant_public` 合并 `TenantProfile`。
- 生命周期状态优先采用明确枚举字段，不继续把全部运营语义塞进 `is_active`。
- 服务到期时间、试用截止时间一期作为“当前状态快照”；续期动作必须写审计日志，但不伪造订单、支付或订阅记录。
- 套餐查询新增类似聚合函数，把 `TenantPlan` 与 `TenantPlanProfile` 合并成对外响应。
- 套餐菜单授权接口复用角色菜单授权的菜单存在性校验模式。
- 套餐同步接口一期允许同步执行；若后续数据量上升，再演进为后台任务。
- 套餐同步的落点是租户内置角色菜单基线，例如 `super_admin`、`admin`、`user`，自定义角色默认不覆盖。

## 11. 数据迁移与兼容策略

### 11.1 数据迁移原则

- 优先复用现有表结构。
- 若当前数据库尚未完全创建扩展表或字段，以 Alembic 迁移补齐。
- 历史记录默认允许扩展字段为空。

### 11.2 存量数据处理

| 对象 | 处理策略 |
| --- | --- |
| 历史租户 | 缺失 `TenantProfile` 时补空白 profile 或查询时兜底 |
| 历史套餐 | 缺失 `TenantPlanProfile` 时补默认 profile |
| 历史套餐菜单 | 缺失 `TenantPlanMenu` 关系时按空菜单处理，再由管理员补授权 |
| 默认套餐 | 初始化时保证存在必要保底菜单 |
| 默认模板 | 持续保持唯一默认模板 |

### 11.3 回滚策略

- 页面级功能可通过隐藏按钮快速回退前端入口。
- 新增接口不影响旧接口主链路，可按路由级回退。
- 扩展字段写入失败时，不应影响已存在的租户主记录查询。

## 12. 风险与应对

| 风险 | 说明 | 应对策略 |
| --- | --- | --- |
| 运营化做成“字段堆砌” | 做完后仍不能支撑试用、续费、冻结管理 | 优先交付生命周期状态、时间、负责人与筛选动作 |
| 套餐授权与角色授权语义混淆 | 容易不清楚最终权限来自哪里 | 文档与页面文案明确“套餐是功能基线，角色是用户授权” |
| 套餐同步覆盖策略不清晰 | 未来如果出现人工例外菜单会有冲突 | 一期先约束为“同步套餐基线”，人工例外能力后置 |
| 扩展字段较多导致表单复杂 | 页面体验可能变重 | 按分组展示，优先展示高频字段 |
| 同步逻辑影响面大 | 错误同步可能影响多个租户菜单 | 增加二次确认、日志、统计反馈，优先支持单套餐同步 |
| 旧数据 profile 缺失 | 页面可能空指针或字段不一致 | 后端统一兜底并补 `ensure_*_profile` 逻辑 |
| 生命周期字段持续膨胀 | 续费、支付、套餐变更历史可能挤入租户资料 | 设置订阅层升级触发条件，达到条件即新增 `TenantSubscription` |

## 13. 订阅层演进预案

一期不实现 `TenantSubscription`，但接口和字段命名不得暗示租户资料就是订单事实。建议预留以下演进路径：

1. 一期：租户保存当前状态、试用截止时间、服务到期时间，运营动作写审计日志。
2. 二期：新增订阅记录，保存套餐、开始/结束时间、订阅状态、价格快照。
3. 三期：订单、支付、退款、账单与订阅关联，租户页只聚合展示当前有效订阅和财务摘要。

启动二期的判定条件：出现多次续费历史查询、套餐升降级、支付状态、退款或财务对账中的任意一项。

## 14. 建议排期顺序

建议按以下顺序实施：

1. 锁定字段与接口范围。
2. 先完成租户生命周期字段与状态流转。
3. 完成后端聚合出参与写入逻辑。
4. 完成套餐菜单授权接口。
5. 完成前端租户与套餐页面增强。

## 17. 模块使用说明

### 17.1 租户管理

1. 通过“新增租户”录入基础信息、生命周期、联系人、经营信息和开通信息。
2. 选择初始化模板后，模板只在租户创建阶段决定基础数据种子；创建后不可通过编辑租户重新执行模板。
3. 通过列表筛选试用、冻结、过期以及 7 天或 30 天内到期租户。
4. 通过行操作执行转正式、续期、冻结、解冻、套餐菜单同步和停用。
5. 通过“运营概览”查看当前配额使用量、只读账务摘要和套餐菜单基线。

### 17.2 套餐管理

1. 套餐表单维护配额、价格、类型、发布状态、排序和展示信息。
2. “菜单授权”维护套餐功能基线；平台管理菜单不可授权，保存时自动补齐所选菜单的祖先节点。
3. 当前默认套餐不可直接编辑、重新授权、停用或删除；其他套餐仍可按原流程设为默认。
4. “同步租户菜单”只同步当前套餐关联的租户。
5. 同步结果返回成功、失败、跳过数量。`super_admin` 跟随套餐基线，`admin`、`user` 保留交集，自定义角色不覆盖。

### 17.3 初始化模板

1. 初始化模板只维护租户开通时的根部门与基础数据种子策略。
2. 默认模板不可取消默认状态、停用或删除。
3. 模板不承担套餐配额、长期菜单授权、续费或账务事实。
