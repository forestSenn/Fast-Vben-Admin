## 2. 实施目标

本计划用于指导当前项目从“功能已基本成形”收口到“v1.0 可发布、可部署、可二次开发”的状态。

当前项目已经具备登录、用户、RBAC、菜单、部门、字典、参数、日志、文件、通知、站内消息、仪表盘、Items、OpenAPI、Docker 等基础能力。后续实施重点不是继续堆新模块，而是把这些模块做成稳定闭环：

- 初始化数据稳定可重复。
- 权限码、菜单、前端按钮和后端接口一致。
- 页面体验一致，关键路径无阻塞问题。
- API 契约稳定，生成类型可用。
- 核心测试可持续运行。
- Docker Compose 最小部署可验证。
- 文档与实际实现一致。

## 3. 总体原则

### 3.1 先收口，后扩展

v1.0 只做后台基座必需能力，不实现岗位、对象存储、OAuth2、多租户、BPM、CRM/ERP/AI。

### 3.2 后端权限是最终边界

所有管理接口必须由后端 `require_permission("<permission_code>")` 控制。前端菜单和按钮隐藏只做体验优化。

### 3.3 每个模块必须闭环

模块完成定义：

- 模型和迁移存在。
- API 可用。
- 权限码接入。
- 默认菜单或入口配置完整。
- 前端页面可操作。
- 测试覆盖核心正向和越权场景。
- 文档更新。

### 3.4 小步验证

每个阶段结束必须运行本阶段验收命令。不得把构建、测试、部署问题留到最后集中处理。

### 3.5 不照搬 Litchi

参考 Litchi 的成熟模块边界和页面组织，但当前项目以 FastAPI + Vben 的轻中量后台基座为目标。Litchi 的平台级能力进入路线图，不进入 v1.0 阻塞范围。

## 15. 风险、阻塞和处理策略

| 风险 | 触发信号 | 处理策略 |
| --- | --- | --- |
| 权限码不一致 | 前端按钮有权限但接口 403，或接口有权限但菜单没有 | M1 先做权限码清单，新增测试 |
| 初始化数据污染 | 重复启动后菜单/角色重复 | 所有种子使用 `ensure_*`，加幂等测试 |
| 动态路由打不开 | 后端 component 和前端文件不匹配 | M1 建 component 映射检查 |
| 操作日志泄密 | 日志出现 password/token | M3 做脱敏测试，敏感路径排除 |
| 文件路径穿越 | 构造 `../` 下载 | 使用 `resolve` 校验并加测试 |
| 前端 build 迟迟失败 | generated 类型和 wrapper 不匹配 | M1 固定 OpenAPI，M5 前不再做破坏性接口 |
| Docker 环境不可复现 | 本地可跑，容器不可跑 | M6 单独做空库 Compose 验收 |
| 后续版本并行膨胀 | 同时启动 Redis、认证和多租户 | 严格按 v1.4、v2.0 顺序设置准入门槛 |

## 18. 后续版本通用规则

### 18.1 状态定义

| 状态 | 判定标准 |
| --- | --- |
| 未开始 | 只有路线图描述，没有模型、迁移、API 和页面 |
| 设计中 | ADR、数据模型或接口契约已评审，但未形成可用闭环 |
| 部分完成 | 已有管理页面或部分 API，但核心业务流程、真实集成或测试缺失 |

### 18.2 跨版本准入规则

- 上一版本 P0 任务未完成，不进入下一版本开发。
- 新增基础设施组件必须提供本地、CI 和 Docker Compose 三种运行方式。
- 外部协议和供应商接入优先采用成熟库或官方 SDK，不自行实现密码学或 OAuth2 协议。
- 新模块必须同时定义权限码、审计策略、敏感字段脱敏和删除保护。
- 所有新列表接口沿用统一分页结构 `items/total/page/page_size`。
- 所有新错误响应沿用统一结构 `code/message/details`。
- 每个版本必须同步更新 OpenAPI 生成结果、README、PRD、TRD、模块指南和 CHANGELOG。

### 20.1 多租户前置 ADR

多租户属于架构改造，开发前必须明确：

- 租户隔离采用共享库共享表、共享库独立 Schema，还是独立数据库。
- 租户识别来自域名、Header、Token claim 还是用户选择。
- 超级管理员、平台管理员和租户管理员的边界。
- 全局数据、租户数据和用户私有数据的分类。
- 唯一索引是否由全局唯一调整为租户内唯一。
- 缓存 key、文件路径、日志和消息如何携带租户上下文。
- 历史数据如何归属默认租户以及如何回滚。
