## 2. 产品结论

本产品首版应定位为“轻量进销存 ERP”，核心是商品、采购、销售、库存和收付款的一体化闭环，而不是覆盖总账、生产制造、人力资源和复杂税务的综合集团 ERP。

ERP v1.0 包含：

- 商品分类、单位、商品、供应商、客户、仓库和结算账户等主数据。
- 采购订单、采购入库、采购退货和付款核销。
- 销售订单、销售出库、销售退货和收款核销。
- 其他入库、其他出库、库存调拨、库存盘点、库存余额和库存流水。
- 销售与采购经营看板、列表查询、详情、导出和审计追踪。
- 租户隔离、模块权益、RBAC、数据权限、RLS、操作审计和并发一致性。

ERP 必须作为独立 `erp` 业务模块交付，不进入 Platform，不扩大中央业务模型，不通过菜单或环境变量临时启用。模块组合由 Edition YAML 显式决定；`erp` 提供独立组合，`suite` 同时包含 Items 与 ERP。

## 3. 背景与机会

当前系统已经具备账号、组织、权限、租户、文件、字典、日志、通知、模块装配和 PostgreSQL 数据隔离能力，但缺少可直接用于真实经营活动的业务套件。参考 Litchi 的 ERP 已形成基础进销存闭环，适合作为功能蓝本。

直接逐页照搬仍存在四个问题：

1. 参考系统的两态审批较轻，必须补充并发、幂等、审计和反审核约束。
2. 参考系统以 Java 单体内部调用组织业务，不能套用到本仓库的模块公开 API 和 Tenant UoW。
3. ERP 数据天然属于租户，不能依赖应用层条件拼接作为唯一隔离手段。
4. “ERP”容易被误解为完整财务或制造系统，需要在产品边界上明确首版是进销存与资金核销。

### 3.1 参考功能映射

| Litchi 参考能力 | 本项目目标 | 适配结论 |
| --- | --- | --- |
| ERP 首页销售/采购统计 | 保留汇总卡片与时间趋势 | 增加租户时区和数据权限口径 |
| 商品、分类、单位 | 完整保留 | 进入 ERP 私有主数据，不放入 Platform |

| 仓库、库存、流水、其他出入库、调拨、盘点 | 完整保留 | 流水设为不可变，反审核使用冲销记录 |
| 结算账户、付款单、收款单 | 完整保留 | 明确仅做源单核销，不宣称具备总账能力 |

### 7.1 页面通用要求

- 列表默认分页，保留筛选条件，支持刷新、列显隐、密度和适用时的导出。
- 单据列表至少支持单号、商品、往来单位、业务日期、制单人、审核状态和备注筛选。
- 主数据表单使用抽屉；包含多行明细的业务单据使用大尺寸抽屉或独立编辑页。
- 单据明细编辑必须支持选商品、批量增行、删除行、即时计算和键盘连续录入。
- “审核”“反审核”“删除”等危险操作必须二次确认，并展示直接业务影响。
- 后端返回 401、403、404、409、422、503 时，前端使用现有错误体系显示可行动提示。
- 小屏设备可查询和查看详情；复杂多行单据允许横向滚动，但主操作不能被遮挡。

## 8. 通用业务规则

### 8.1 租户与数据归属

- 所有 ERP 实体均为 tenant-owned，`tenant_id` 只能来自已验证 JWT 对应的当前 Tenant UoW。
- API 请求体、query 或普通 header 不接受可覆盖数据归属的 `tenant_id`。
- ERP 使用独立 schema、迁移 namespace 和版本表；所有租户表启用对应 RLS policy。
- 商品、客户、供应商、仓库、账户和业务单据的关联必须属于同一租户。
- 列表、详情、更新、删除、审核、反审核和导出应用完全一致的数据权限范围。
- Platform 用户、部门等主数据只能通过 Platform 公开端口访问，ERP 不导入 Platform ORM。

### 8.2 主数据状态

- 主数据状态为 `ENABLED` 或 `DISABLED`。
- 禁用数据保留历史引用和详情展示，但不可用于新建或修改业务单据。
- 已被单据引用的主数据默认禁止物理删除；允许停用。
- 商品分类为树结构，存在下级分类或商品时禁止删除。
- 同一租户最多一个默认仓库、一个默认结算账户；切换默认项必须原子完成。

### 8.3 单据状态

| 当前状态 | 操作 | 目标状态 | 允许条件 | 业务副作用 |
| --- | --- | --- | --- | --- |
| `DRAFT` | 修改 | `DRAFT` | 有更新权限 | 重算表头汇总，不产生库存或核销 |
| `DRAFT` | 删除 | 已删除 | 无下游记录 | 删除表头和明细，保留操作审计 |
| `DRAFT` | 审核 | `APPROVED` | 有审核权限且校验全部通过 | 同事务产生库存、累计数量或核销副作用 |
| `APPROVED` | 反审核 | `DRAFT` | 有反审核权限且不存在阻断下游 | 以反向流水/反向核销抵消，不删除历史流水 |
| `APPROVED` | 修改/删除 | 不允许 | 无 | 返回稳定冲突错误 |

审核和反审核必须带版本条件或等价并发控制。重复提交、并发审核或网络重试只能有一个请求成功产生业务副作用。

### 8.4 单号规则

系统生成单号，不允许客户端自行指定。目标格式为 `{业务前缀}{yyyyMMdd}{6位日序号}`，并在租户内唯一。

| 单据 | 前缀 | 示例 |
| --- | --- | --- |
| 其他入库 | `QTRK` | `QTRK20260720000001` |
| 其他出库 | `QTCK` | `QTCK20260720000001` |
| 库存调拨 | `KCDB` | `KCDB20260720000001` |
| 库存盘点 | `KCPD` | `KCPD20260720000001` |
| 采购订单/入库/退货 | `CGDD` / `CGRK` / `CGTH` | `CGDD20260720000001` |
| 销售订单/出库/退货 | `XSDD` / `XSCK` / `XSTH` | `XSDD20260720000001` |
| 付款/收款 | `FKD` / `SKD` | `FKD20260720000001` |

实现不能依赖 Redis 才能保证唯一性。PostgreSQL 唯一约束是最终边界；序号生成器在冲突时有限重试，失败时整个事务回滚。

### 8.5 数量与金额

- 数量使用十进制定点数，目标精度 `numeric(20, 6)`；必须大于 0，盘点差异除外。
- 单价、金额使用十进制定点数，目标精度 `numeric(20, 4)`；页面金额默认展示 2 位。
- 百分比范围为 0 至 100，使用十进制定点数。
- 行商品金额 = 数量 x 单价；行税额 = 行商品金额 x 税率；行价税合计 = 行商品金额 + 行税额。
- 表头商品金额、税额和总数量由后端根据有效明细重算，不信任客户端汇总字段。
- 订单结算金额 = 商品金额 + 税额 - 优惠金额。
- 入库/退货结算金额 = 商品金额 + 税额 - 优惠金额 + 其他费用。
- 销售出库/销售退货结算金额 = 商品金额 + 税额 - 优惠金额 - 其他扣减。
- 舍入方式、精度和计算顺序必须全局固定；同一公式用于创建、更新、详情、导出和统计。

### 8.6 库存记账

- 库存余额以 `(tenant_id, product_id, warehouse_id)` 唯一。
- 所有库存变化必须通过库存记账服务执行，不允许业务 Repository 直接改余额。
- 审核库存类单据时，在同一数据库事务内完成状态变更、余额原子增减和库存流水写入。
- 默认不允许负库存。销售出库、其他出库、采购退货、调拨出库和盘亏导致余额小于 0 时失败。
- 库存流水不可修改、不可删除；反审核写入具有相反数量的冲销流水，并关联原单据及原单据行。
- 流水保存变动数量、变动后余额、业务类型、业务单 ID、业务行 ID、业务单号、操作人和时间。
- 调拨审核必须同时写调出负流水与调入正流水，任一失败则全部回滚。
- 库存余额和流水聚合必须提供内部对账检查；发现差异时模块健康状态进入 degraded，禁止继续产生不可信库存。

## 9. 商品管理需求

### 9.1 商品单位 `ERP-PROD-UNIT`

字段：单位名称、状态、创建人、创建时间、更新时间。

功能与验收：

- 支持列表、创建、编辑、停用、删除和导出。
- 同一租户单位名称唯一，空白和首尾空格归一化后仍需唯一。
- 已被商品引用时禁止删除；停用后历史单据仍展示单位快照。

### 9.2 商品分类 `ERP-PROD-CATEGORY`

字段：上级分类、分类名称、分类编码、排序、状态。

功能与验收：

- 支持树形查询、创建、编辑、移动、停用、删除和导出。
- 同一租户分类编码唯一；禁止将节点移动到自身或其后代。
- 存在子分类或商品时禁止删除。
- 停用父分类后，该分类及其后代不可用于新商品选择。

### 9.3 商品资料 `ERP-PROD-PRODUCT`

| 字段 | 必填 | 规则 |
| --- | --- | --- |
| 商品名称 | 是 | 租户内按产品策略校验重名，去除首尾空格 |
| 条码 | 否 | 租户内非空唯一 |
| 分类 | 是 | 必须为当前租户启用的末级或允许挂载商品的分类 |
| 单位 | 是 | 必须启用 |
| 规格 | 否 | 文本描述，v1.0 不拆多 SKU |
| 采购参考价 | 否 | 大于等于 0 |
| 销售参考价 | 否 | 大于等于 0 |
| 最低销售价 | 否 | 大于等于 0，默认不得高于销售参考价 |
| 重量 | 否 | 大于等于 0 |
| 保质天数 | 否 | 大于等于 0；v1.0 只记录，不做批次预警 |
| 状态 | 是 | 启用/停用 |
| 备注 | 否 | 长度受限并防止脚本注入 |

商品列表支持名称、条码、分类、状态筛选和导出。停用商品不可加入新单据。商品被业务单据引用后禁止物理删除。

## 10. 往来单位需求

### 10.1 供应商 `ERP-PUR-SUPPLIER`

字段：名称、联系人、手机、电话、邮箱、传真、税号、默认税率、开户行、银行账号、开户地址、排序、状态、备注。

验收要求：

- 名称在租户内唯一；邮箱、手机号和税率格式合法。
- 禁用供应商不可用于新采购订单、其他入库或付款单。
- 已存在采购或付款数据时禁止删除。
- 银行账号在列表默认脱敏，只有具备敏感财务信息权限的用户可查看完整值。

### 10.2 客户 `ERP-SALE-CUSTOMER`

字段与供应商一致，语义用于销售和收款。

验收要求：

- 禁用客户不可用于新销售订单、其他出库或收款单。
- 已存在销售或收款数据时禁止删除。
- 银行账号、联系方式按权限和导出策略脱敏。

## 11. 采购管理需求

### 11.1 采购订单 `ERP-PUR-ORDER`

表头字段：单号、供应商、订单时间、结算账户、优惠率、优惠金额、订金、附件、备注、审核状态、总数量、商品金额、税额、结算金额、累计入库数量、累计退货数量、制单人和审计时间。

明细字段：商品、单位、数量、采购单价、商品金额、税率、税额、价税合计、累计入库数量、累计退货数量和备注。

业务规则：

- 新建时至少一行明细，同一商品允许多行但必须有明确业务需要；默认提示合并。
- 供应商、商品、单位必须启用，价格和税率由商品/供应商默认值带出但可在权限允许时修改。
- 审核后订单可被采购入库和采购退货选单使用。
- 单行累计入库数量不得超过订单数量。
- 单行累计退货数量不得超过已入库数量；未发生入库不得从订单直接退货。
- 已存在有效入库或退货记录时禁止反审核订单。
- 入库状态和退货状态按无、部分、全部实时派生，不由客户端写入。

### 11.2 采购入库 `ERP-PUR-IN`

表头字段：单号、来源采购订单、供应商快照、入库时间、结算账户、优惠率、其他费用、附件、备注、审核状态、总数量、商品金额、税额、结算金额、已付款和未付款金额。

明细字段：来源订单行、仓库、商品、单位、数量、单价、税率、金额和备注。

业务规则：

- 只能选择已审核且仍有可入库数量的采购订单。
- 明细商品和单位继承来源订单；入库数量不得超过订单行剩余可入库数量。
- 审核后增加对应仓库库存，写采购入库流水，并更新订单累计入库数量。
- 反审核时减少库存并写冲销流水；若库存不足以冲销，反审核失败。
- 已存在有效付款核销时禁止反审核。
- 付款状态按未付款、部分付款、已付清派生。

### 11.3 采购退货 `ERP-PUR-RETURN`

表头字段与采购入库类似，业务时间为退货时间，金额字段包含已冲减和未冲减金额。为兼容参考产品，界面可使用“已退款/未退款”文案，但其 v1.0 业务含义是对供应商应付款进行负向冲减。

业务规则：

- 只能选择已审核且存在可退数量的采购订单及其已入库商品。
- 退货仓库必须拥有足够库存；退货数量不得超过订单行累计已入库减累计已退数量。
- 审核后扣减库存、写采购退货流水并更新订单累计退货数量。
- 反审核后恢复库存并写冲销流水；已存在有效付款净额冲减时禁止反审核。
- 冲减状态按未冲减、部分冲减、已冲减派生。

## 12. 销售管理需求

### 12.1 销售订单 `ERP-SALE-ORDER`

表头字段：单号、客户、销售员、订单时间、结算账户、优惠率、优惠金额、订金、附件、备注、审核状态、总数量、商品金额、税额、结算金额、累计出库数量、累计退货数量和审计字段。

明细字段：商品、单位、数量、销售单价、商品金额、税率、税额、价税合计、累计出库、累计退货和备注。

业务规则：

- 客户、销售员、商品和单位必须有效；低于商品最低销售价时默认拒绝，后续可增加独立越权审批能力。
- 审核后才可创建销售出库和销售退货。
- 单行累计出库不得超过订单数量。
- 单行累计退货不得超过累计已出库数量。
- 已存在有效出库或退货记录时禁止反审核。
- 出库和退货状态按无、部分、全部派生。

### 12.2 销售出库 `ERP-SALE-OUT`

表头字段：单号、来源销售订单、客户快照、销售员、出库时间、结算账户、优惠率、其他扣减、附件、备注、审核状态、总数量、结算金额、已收款和未收款金额。

明细字段：来源订单行、仓库、商品、单位、可用库存、数量、单价、税率、金额和备注。

业务规则：

- 只能选择已审核且仍有可出库数量的销售订单。
- 审核时逐行校验库存；任何一行不足则整单失败，不允许部分写入。
- 审核后扣减库存、写销售出库流水并更新订单累计出库数量。
- 反审核后恢复库存并写冲销流水；已存在有效收款核销时禁止反审核。
- 收款状态按未收款、部分收款、已收清派生。

### 12.3 销售退货 `ERP-SALE-RETURN`

表头和明细结构与销售出库对应，业务时间为退货时间，金额字段包含已冲减和未冲减金额。为兼容参考产品，界面可使用“已退款/未退款”文案，但其 v1.0 业务含义是对客户应收款进行负向冲减。

业务规则：

- 只能退已审核销售订单中已经出库且尚未退回的数量。
- 审核后增加退回仓库库存、写销售退货流水并更新订单累计退货数量。
- 反审核时扣减库存；库存不足时失败。
- 已存在有效收款净额冲减时禁止反审核。

## 13. 库存管理需求

### 13.1 仓库 `ERP-STOCK-WAREHOUSE`

字段：名称、地址、负责人、仓储费用参考值、运输费用参考值、排序、状态、是否默认、备注。

要求：名称租户内唯一；只能有一个默认仓库；停用仓库不可参与新库存单据；存在非零库存或历史单据时禁止删除。

### 13.2 库存余额 `ERP-STOCK-BALANCE`

- 只读列表，按商品、分类、仓库筛选和导出。
- 展示商品、分类、单位、仓库、当前数量和更新时间。
- 不提供直接编辑数量入口；修正库存必须使用其他入库、其他出库或盘点单。
- 具备详情跳转，可查看当前商品/仓库对应库存流水。

### 13.3 库存流水 `ERP-STOCK-LEDGER`

业务类型包括：其他入库/冲销、其他出库/冲销、调拨入库/出库及冲销、盘盈/盘亏及冲销、采购入库/冲销、采购退货/冲销、销售出库/冲销、销售退货/冲销。

列表支持商品、仓库、业务类型、业务单号和时间范围筛选，展示变动数量、变动后余额、来源单号、来源行、操作人和时间。用户可以跳转到有权访问的来源单据；无来源权限时不泄露详情。

### 13.4 其他入库与其他出库 `ERP-STOCK-OTHER`

表头字段：单号、业务时间、可选供应商/客户、附件、备注、状态、总数量和总金额。

明细字段：仓库、商品、单位、当前库存、数量、参考单价、金额和备注。

审核其他入库增加库存；审核其他出库减少库存。反审核写反向流水。已审核单据不可编辑或删除。

### 13.5 库存调拨 `ERP-STOCK-MOVE`

表头字段：单号、调拨时间、附件、备注、状态、总数量和总金额。

明细字段：调出仓库、调入仓库、商品、单位、调出可用库存、数量、参考单价、金额和备注。

要求：调出和调入仓库不得相同；调出库存必须充足；审核后同事务完成两端记账；反审核时调入仓库库存必须足够扣回。

### 13.6 库存盘点 `ERP-STOCK-CHECK`

表头字段：单号、盘点时间、附件、备注、状态、差异总数量和差异总金额。

明细字段：仓库、商品、单位、账面数量、实盘数量、差异数量、参考单价、差异金额和备注。

业务规则：

- 创建时读取账面数量并保存快照；实盘数量必须大于等于 0。
- 差异数量 = 实盘数量 - 账面快照，零差异行允许保留但审核时不记流水。
- 审核前再次检查当前账面数量是否仍等于快照；发生并发库存变化时返回冲突并要求刷新重盘。
- 正差异写盘盈入库，负差异写盘亏出库；反审核写对应冲销流水。

## 14. 财务管理需求

ERP v1.0 的财务能力只负责结算账户与业务源单核销，不形成会计凭证或总账。

### 14.1 结算账户 `ERP-FIN-ACCOUNT`

字段：账户名称、账号/内部编号、排序、状态、是否默认、备注。

要求：名称和非空编号租户内唯一；只能有一个默认账户；禁用账户不可用于新单据；有核销历史时禁止删除。完整账号按财务敏感权限控制展示和导出。

### 14.2 付款单 `ERP-FIN-PAYMENT`

表头字段：单号、付款时间、财务人员、供应商、结算账户、源单总额、优惠金额、实付金额、备注和审核状态。

核销明细：业务类型、来源采购入库/采购退货、源单号、有符号源单金额、历史已核销、当前有符号核销金额和备注。

业务规则：

- 同一付款单的核销明细必须属于同一供应商和当前租户。
- 仅可选择已审核且仍有未结金额的源单。采购入库是正向应付，采购退货是负向应付冲减；符号由后端根据业务类型确定，客户端不能覆盖。
- 采购入库行的当前付款金额大于 0 且不得超过剩余应付；采购退货行的冲减绝对值不得超过剩余可冲减金额。
- 单据实付金额 = 正向付款合计 - 退货冲减合计 - 优惠金额，结果不得小于 0。结果为 0 时属于非现金净额结算，必须在页面明确标识。
- 审核付款单后更新源单累计付款/冲减金额；反审核以反向核销恢复剩余金额。
- 已审核付款单不可修改或删除；并发核销同一源单时只能一个事务占用剩余额度。

### 14.3 收款单 `ERP-FIN-RECEIPT`

表头字段与付款单对应，往来单位为客户，核销来源为销售出库/销售退货。销售出库是正向应收，销售退货是负向应收冲减，符号由后端确定。

业务规则与付款单对称：正向收款不得超过剩余应收，退货冲减绝对值不得超过剩余可冲减金额，单据实收净额不得小于 0，审核和反审核原子更新源单累计金额。

### 14.4 结算口径

- 采购入库形成应付，采购退货形成供应商信用额，在付款单内抵减同一供应商应付。
- 销售出库形成应收，销售退货形成客户信用额，在收款单内抵减同一客户应收。
- ERP v1.0 使用用户明确选择的源单级净额核销，不自动跨单匹配。
- 当供应商信用额大于可抵应付、或客户信用额大于可抵应收时，剩余信用额继续保持未核销。独立供应商退款入账和客户现金退款不在当前两类单据中隐式处理。
- 订金在 v1.0 只作为订单业务字段；是否自动转核销必须在开发前由产品负责人确认，未确认时不得自动影响收付款余额。

## 15. ERP 首页与统计

### 15.1 汇总卡片

- 今日销售额、昨日销售额、本月销售额、今年销售额。
- 今日采购额、昨日采购额、本月采购额、今年采购额。
- 销售额 = 已审核销售出库金额 - 已审核销售退货金额。
- 采购额 = 已审核采购入库金额 - 已审核采购退货金额。

### 15.2 时间趋势

- 默认展示近期时间序列的销售额与采购额，粒度根据范围切换为日/月。
- 所有统计使用租户时区划分自然日，金额使用与单据相同的舍入口径。
- 反审核后统计应随有效单据状态更新，不重复累计冲销流水。
- 首页数据权限与单据查询一致；无全租户数据权限时明确标注“我的数据范围”。

## 16. 权限设计

### 16.1 权限命名

业务入口统一使用 `require_module_access("erp", "<permission>")`。权限码格式为 `erp:<resource>:<action>`。

| 资源组 | 示例权限 |
| --- | --- |
| 首页 | `erp:statistics:query` |
| 商品 | `erp:product:list/create/update/delete/export` |
| 分类/单位 | `erp:product-category:*`、`erp:product-unit:*` |
| 供应商/客户 | `erp:supplier:*`、`erp:customer:*` |
| 采购单据 | `erp:purchase-order:*`、`erp:purchase-in:*`、`erp:purchase-return:*` |
| 销售单据 | `erp:sale-order:*`、`erp:sale-out:*`、`erp:sale-return:*` |
| 库存 | `erp:warehouse:*`、`erp:stock:list/export`、`erp:stock-record:list/export` |
| 库存单据 | `erp:stock-in:*`、`erp:stock-out:*`、`erp:stock-move:*`、`erp:stock-check:*` |
| 财务 | `erp:account:*`、`erp:finance-payment:*`、`erp:finance-receipt:*` |

单据 `*` 至少拆分为 `list`、`create`、`update`、`delete`、`approve`、`reverse`、`export`。审核与反审核必须是独立权限，不能沿用普通更新权限。

### 16.2 权限验收

- 前端菜单和按钮显隐与后端权限码一致，但后端始终是最终安全边界。
- 无列表权限的用户不能通过详情 ID、导出、选单弹窗或统计接口旁路读取数据。
- 无财务敏感权限时，银行账号、结算账号等字段在详情和导出中一致脱敏。
- 超级管理员绕过只发生在 Platform 授权层；ERP Repository 不自行判断超级管理员。

## 17. 数据模型边界

### 17.1 ERP 拥有的数据

| 类别 | 目标实体 |
| --- | --- |
| 商品 | `ProductCategory`、`ProductUnit`、`Product` |
| 往来单位 | `Supplier`、`Customer` |
| 库存 | `Warehouse`、`StockBalance`、`StockLedger` |
| 采购 | `PurchaseOrder` 与 `PurchaseOrderItem`、`PurchaseIn` 与 `PurchaseInItem`、`PurchaseReturn` 与 `PurchaseReturnItem` |
| 销售 | `SaleOrder` 与 `SaleOrderItem`、`SaleOut` 与 `SaleOutItem`、`SaleReturn` 与 `SaleReturnItem` |
| 库存单据 | `StockIn` 与 `StockInItem`、`StockOut` 与 `StockOutItem`、`StockMove` 与 `StockMoveItem`、`StockCheck` 与 `StockCheckItem` |
| 财务 | `SettlementAccount`、`FinancePayment` 与 `FinancePaymentItem`、`FinanceReceipt` 与 `FinanceReceiptItem` |

所有实体表名、约束、索引、RLS policy、授权和 downgrade 行为必须由 ERP 自己的 Alembic namespace 管理。

### 17.2 跨模块引用

- ERP 可以保存 Platform 用户、部门等稳定 ID，但不建立跨模块数据库外键。
- 用户显示名等易变化字段在查询时通过 Platform 公开查询端口批量解析；关键历史单据可保存必要快照。
- Platform 删除用户、租户或其他主数据前，通过 ERP `ReferenceGuardSpec` 检查引用。
- 未来 CRM、BPM 或其他模块只能通过 ERP `public_api` DTO、命令、查询和版本化事件协作，不得查询 ERP ORM。

### 17.3 目标事件

| 事件 | 版本 | 触发时机 | 关键数据 |
| --- | --- | --- | --- |
| `erp.purchase_order.approved` | v1 | 采购订单首次审核成功 | tenant、order_id、no、supplier_id、total_price |
| `erp.purchase_in.approved` | v1 | 采购入库审核成功 | tenant、document_id、order_id、stock effects |
| `erp.sale_order.approved` | v1 | 销售订单首次审核成功 | tenant、order_id、no、customer_id、total_price |
| `erp.sale_out.approved` | v1 | 销售出库审核成功 | tenant、document_id、order_id、stock effects |
| `erp.stock.changed` | v1 | 库存事务提交 | tenant、product_id、warehouse_id、delta、balance、source |
| `erp.payment.approved` | v1 | 付款审核成功 | tenant、payment_id、supplier_id、amount |
| `erp.receipt.approved` | v1 | 收款审核成功 | tenant、receipt_id、customer_id、amount |

反审核使用对应明确事件或事件动作字段，不能把“审核事件金额取负”作为未声明约定。事件和业务状态使用事务 Outbox 同时提交，消费者按 event ID 幂等处理。

## 18. API 与错误契约要求

- API 前缀目标为 `/api/v1/erp`，按资源分组；OpenAPI 是前端类型的唯一来源。
- 创建接口支持 `Idempotency-Key` 或等价幂等键；审核、反审核和导入类命令必须防重复提交。
- 分页统一使用当前项目规范；筛选时间使用带时区时间，服务端按租户时区解释自然日。
- 错误返回稳定 `code` 与安全 `message`，至少区分不存在、已停用、状态冲突、库存不足、超量、超额核销、跨租户、无权限和模块不可用。
- 409 用于并发版本、重复审核、下游依赖阻断、库存不足和额度被并发占用等业务冲突。
- 422 用于字段格式、必填、数值范围和明细结构错误。
- 不能向客户端返回 SQL、堆栈、内部表名、租户数据或第三方 SDK 对象。

## 19. 导入、导出、附件与审计

### 19.1 导入导出

- v1.0 P0 必须支持各列表导出；导出应用当前筛选、数据权限和字段脱敏。
- 大数据导出应采用受控后台任务或流式响应，不能一次加载全部记录。
- 商品、客户、供应商批量导入列为 P1；提供版本化模板、逐行校验结果和部分成功策略说明。
- 业务单据批量导入不进入 v1.0，避免绕过选单、库存和审核规则。

### 19.2 附件

- 单据附件复用 Platform 文件公开能力，ERP 只保存稳定文件引用，不导入存储实现。
- 上传、预览、下载和删除均校验租户、单据数据范围和文件权限。
- 已审核单据附件默认不可替换；如允许补充附件，必须记录独立审计事件。

### 19.3 审计

- 创建、修改、删除、审核、反审核、导出和查看敏感财务字段均记录操作日志。
- 审核日志至少记录操作人、时间、来源 IP、原状态、目标状态、单据版本和失败原因。
- 业务单据保留创建人、更新人、审核人、审核时间和版本号。
- 密码、token、完整银行账号及文件签名 URL 不进入日志明文。

## 20. 非功能需求

### 20.1 安全与隔离

- ERP 所有 tenant-owned 表在 `app_runtime` 角色下验证 RLS 读写、bulk DML、无 scope 和跨租户负向场景。
- `app_runtime` 无超级用户、建库、建角色或 `BYPASSRLS` 权限。
- 金额、数量和权限校验全部在服务端重复执行，不能信任前端只读字段。
- 导出、选择器、统计和健康检查不得形成越权旁路。

### 20.2 一致性与恢复

- 单据状态、明细、库存余额、库存流水、订单累计量、核销累计额和 Outbox 必须按业务原子性放在同一事务。
- 任何事务失败都不得留下半张单据、半边调拨或只变余额未写流水的状态。
- 数据库唯一约束、检查约束和并发更新条件共同保护关键不变量。
- 提供库存余额与流水、源单金额与核销明细的对账命令；只报告差异，不自动无审计修复。

### 20.3 性能

- 所有列表强制分页，默认每页 20，最大页大小遵循平台限制。
- 高频筛选字段建立租户前缀复合索引，包括单号、业务日期、状态、往来单位、商品和仓库。
- 选单查询只返回仍有可操作余量的已审核单据，并支持服务端搜索。
- 首页统计允许短时缓存，但缓存键必须包含租户和数据范围；审核/反审核后按一致策略失效。

### 20.4 可用性与可观测性

- ERP 模块未安装、迁移落后、依赖不可用或对账异常时，健康状态必须可区分。
- 关键指标至少包括审核成功/失败数、库存不足冲突数、单据处理耗时、Outbox backlog 和对账差异数。
- 错误日志包含 trace ID、租户安全标识和业务单号，但不包含敏感正文。

## 21. Edition 与模块装配目标

Edition 组合如下，具体模块集合以 `editions/<edition>.yaml` 为唯一事实源：

| Edition | 目标模块 |
| --- | --- |
| `base` | `platform` |
| `items` | `platform`、`items` |
| `erp` | `platform`、`erp` |
| `suite` | `platform`、`items`、`erp` |

ERP `ModuleDefinition` 负责声明 Router、权限、菜单、迁移、事件、引用保护和生命周期组件。前端代码进入 `frontend/apps/web-antd/src/modules/erp`，只通过生成的 Build Manifest 注册组件和客户端。Base/Items 构建不得包含或预加载 ERP 运行入口。

## 22. 交付阶段

### 22.1 Phase A：模块基础与库存底座

范围：ERP ModuleDefinition、Edition、独立 schema/migration、商品分类/单位/商品、仓库、库存余额/流水、其他入库/出库、调拨、盘点、权限、RLS 和审计。

退出标准：所有库存入口均通过统一记账服务；审核、反审核、并发、负库存和跨租户测试通过；余额与流水可对账。

### 22.2 Phase B：采购与销售闭环

范围：供应商、客户、采购订单/入库/退货、销售订单/出库/退货、选单组件、累计数量和状态派生。

退出标准：支持分批入库/出库/退货；超量、未审核源单、禁用主数据、下游阻断反审核等负向场景通过。

### 22.3 Phase C：财务核销与经营首页

范围：结算账户、付款单、收款单、核销、销售/采购汇总与趋势、导出和完整前端体验。

退出标准：并发核销不超额，反审核正确恢复余额；统计口径与已审核单据明细对账一致；ERP v1.0 Edition 完整构建。

各 Phase 可以独立合并技术增量，但租户权益默认关闭；没有完成 P0 闭环和数据迁移验证前，不对外宣称 ERP 可交付。

## 23. 验收场景

### 23.1 采购到付款

1. 创建并审核 100 件商品的采购订单。
2. 分两次入库 40 件和 60 件，库存累计增加 100，订单显示全部入库。
3. 再尝试入库 1 件，系统返回超量冲突且库存不变。
4. 创建采购退货 10 件并审核，库存减少 10，订单累计退货为 10。
5. 对两张采购入库分别部分付款，已付与未付金额准确。
6. 对已核销入库单执行反审核，系统拒绝并说明存在付款记录。

### 23.2 销售到收款

1. 在库存 90 件时创建并审核销售订单 80 件。
2. 分批出库 30 件和 50 件，库存降至 10，订单显示全部出库。
3. 尝试额外出库或其他出库 11 件，整单失败且无部分流水。
4. 销售退货 5 件，库存恢复至 15。
5. 创建部分收款和尾款，源单从未收转为部分收款再转为已收清。

### 23.3 库存一致性

1. 调拨 5 件时同时产生调出、调入两条流水，两仓总库存不变。
2. 盘点创建后发生其他出库，再审核原盘点单，系统提示账面快照过期。
3. 两个用户并发审核同一出库单，只能产生一次库存扣减。
4. 反审核库存单据后保留原流水，并新增相反数量的冲销流水。
5. 对账命令聚合流水后与库存余额完全一致。

### 23.4 权限与租户

1. 未启用 ERP 权益的租户访问任意 ERP API 返回模块访问错误。
2. 有菜单但无审核权限的用户无法通过直接请求审核单据。
3. 用户无法通过替换 ID 查看、导出或选择其他租户的数据。
4. 无租户 scope 的 `app_runtime` 查询不到 ERP tenant-owned 数据且不能写入。
5. 本部门销售员无法在详情、统计或导出中旁路读取其他部门订单。

## 25. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 将轻量进销存误称完整 ERP | 需求持续膨胀 | 固定 v1.0 非目标范围，新增域单独立项 |
| 并发审核导致重复记账 | 库存和资金失真 | 条件更新、幂等键、唯一约束和事务测试 |
| 余额字段与明细漂移 | 订单状态和核销错误 | 派生字段统一服务更新并提供对账检查 |
| 反审核删除历史流水 | 审计链断裂 | 强制使用冲销流水，不允许修改或删除流水 |
| 数据权限只保护列表 | 详情或导出越权 | 所有入口复用同一 scope builder 并做负向测试 |
| ERP 静态依赖 Platform ORM | 模块边界失效 | 只依赖 Platform 公开端口和稳定 DTO |
| 一次性交付范围过大 | 长期不可用 | 按库存底座、进销闭环、财务统计三阶段交付 |
| 金额口径含糊 | 对账争议 | 开发前签署税额、优惠、其他费用和订金口径 |

## 26. 开发前待确认事项

以下事项不影响本 PRD 确立总体范围，但必须在对应 Phase 开发前由产品负责人确认：

1. 商品名称是否允许租户内重名，还是只要求条码唯一。
2. 商品分类是否只允许末级分类挂商品。
3. 销售价低于最低销售价是直接禁止，还是增加独立越权权限。
4. 采购/销售订单的订金是否在 v1.0 自动进入核销。
5. “其他费用”在销售出库和销售退货中的业务名称及正负方向。
6. 采购退货是否必须严格来源于采购订单，还是允许来源于已审核采购入库单。
7. 业务日期是否允许回填历史日期，以及跨结账期是否需要限制。
8. 导出文件是否需要 Excel，还是首版允许 CSV。
9. Phase C 上线前是否需要商品、客户和供应商批量导入。
10. 是否在 v1.0 增加独立的供应商退款收款与客户退款付款流程；若不增加，则只支持退货信用额抵减同一往来单位的未结款。

未确认项必须采用失败关闭或本 PRD 给出的保守默认，不得在实现中静默形成不可逆业务规则。

## 27. ERP v1.0 完成定义

只有同时满足以下条件，才能宣称 ERP v1.0 完成：

- 第 6.1 节所有 P0 能力形成可操作闭环，不存在占位页面或旁路写入。
- ERP Edition、ModuleDefinition、独立迁移、Manifest 和前端注册均可重现。
- 所有 tenant-owned 表完成 Tenant UoW、自动过滤、写入校验、RLS 和跨租户负向验证。
- 审核、反审核、库存记账、累计数量、核销金额和 Outbox 事务一致。
- 库存余额与流水、源单金额与核销明细对账为零差异。
- 权限、数据范围、敏感字段、附件和导出无已知越权路径。
- 后端 lint/测试、迁移检查、前端 typecheck、ERP E2E 和受影响 Edition 构建实际通过。
- 兼容、迁移、升级、回退、监控和运维文档与实现一致。
- 未执行的远端 CI、镜像扫描、签名、N-1 演练或团队签署被明确列出，不冒充已完成。
