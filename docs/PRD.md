## 2. 背景与结论

当前项目已经从基础模板演进到可用后台基座：后端已有登录认证、用户、角色、菜单、部门、岗位、字典、系统参数、日志、文件、通知、站内消息、仪表盘和示例业务模块；前端 `web-antd` 也已经完成动态权限接入，并把参数配置、文件管理沉淀为顶层“基础设施”菜单。它已经不再只是 `full-stack-fastapi-template + vue-vben-admin` 的简单拼接。

`LZ-litchi` 和 `LZ-litchi-ui-admin-vben` 的价值在于给出了一个成熟快速开发平台的完整功能版图：系统管理、基础设施、权限菜单、组织岗位、租户、国际化、代码生成、文件服务、监控、BPM、CRM、ERP、AI 等。但这些能力不能直接照搬到当前项目，否则会让 FastAPI 版本在 v1.0 前失去焦点。

本 PRD 的核心结论是：

- v1.0 定位为“开箱即用的 Python 中后台基座”，不是低代码平台、BPM 平台或业务套件。
- v1.0 必须把当前已实现的通用后台能力打磨完整，形成可二次开发、可部署、可验收的闭环。
- Litchi 中的多租户、OAuth2/SSO、岗位、代码生成、监控、BPM 等能力进入 v1.x 路线图，不作为 v1.0 阻塞项。
- 当前项目需要补强的不是“堆更多菜单”，而是数据初始化、权限一致性、页面体验、测试、文档和部署可验证性。

## 3. 产品定位

Fast Vben Admin 是面向 Python/Vue 技术栈团队的开箱即用中后台系统基座。它提供账号、组织、权限、菜单、字典、参数、日志、文件、通知、仪表盘和示例业务模块，使开发者可以直接在其上开发内部管理系统、SaaS 管理端、数据运营后台和项目交付后台。

### 3.1 一句话愿景

让开发者拿到项目当天就能开始写业务，而不是先花一周补账号、权限、菜单、日志、上传、通知和部署这些通用基建。

### 3.2 差异化

| 维度 | Fast Vben Admin | LZ-litchi |
| --- | --- | --- |
| 后端技术栈 | FastAPI、SQLModel、Pydantic、PostgreSQL | Spring Boot、MyBatis Plus、Redis、多数据库 |
| 目标用户 | Python/Vue 团队、轻中量后台、二次开发项目 | Java 企业级快速开发平台、复杂业务套件 |
| v1.0 目标 | 后台基座完整闭环 | 平台化、模块化、生态化 |
| 功能策略 | 先做通用必需能力 | 覆盖系统、基础设施、BPM、CRM、ERP、AI |
| 产品边界 | 不做复杂低代码、BPM、多租户 | 包含更多平台级能力 |

## 6. 产品目标

### 6.1 v1.0 目标

v1.0 的目标是达到“可直接二次开发和小型项目交付”的实用程度。

用户应能做到：

1. 克隆项目后按 README 在 10 分钟内启动完整系统。
2. 使用默认管理员登录后台。
3. 创建部门、角色、用户，并分配菜单和按钮权限。
4. 通过后端权限控制接口访问，普通用户无法越权。
5. 管理字典、系统参数、通知公告和站内消息。
6. 查看登录日志、操作日志，并追踪关键操作。
7. 上传、查看、下载和删除文件。
8. 参考 Items 模块新增一个完整业务模块。
9. 通过 OpenAPI 生成前端类型，保持前后端契约一致。
10. 使用 Docker Compose 完成最小生产部署。

### 6.2 成功指标

| 指标 | v1.0 目标 |
| --- | --- |
| 首次启动时间 | 小于 10 分钟 |
| 默认后台可用性 | 登录后无需改代码即可使用系统管理核心功能 |
| 通用能力覆盖 | 覆盖用户、角色、菜单、部门、字典、参数、日志、文件、通知 |
| 权限安全 | 无权限菜单不可见，无权限接口返回 403 |
| 新模块开发效率 | 开发者可按文档在 1 小时内新增简单 CRUD |
| 测试质量 | 后端核心测试、前端构建、关键 E2E 通过 |
| 部署体验 | Compose 可完成单机最小生产部署 |

## 8. v1.0 功能需求

### 8.1 认证与账号

目标：提供安全、完整、可用于中后台的账号闭环。

功能：

- 邮箱 + 密码登录。
- JWT access token。
- 退出登录。
- 获取当前用户。
- 更新个人资料。
- 修改密码。
- 忘记密码和重置密码。
- 默认超级管理员初始化。
- 禁用用户不可登录。
- 登录日志记录成功和失败。

验收标准：

- 默认管理员可以登录。
- 禁用用户无法登录和访问接口。
- 修改密码后旧密码失效。
- 重置密码 token 有过期时间。
- 登录失败不暴露账号是否存在等敏感细节。

### 8.2 用户管理

目标：管理员可维护系统用户，并建立用户、部门、角色关系。

功能：

- 用户列表，支持分页、搜索、状态筛选。
- 新增、编辑、删除用户。
- 启用/禁用用户。
- 重置用户密码。
- 分配部门。
- 分配角色。
- 标识超级管理员。
- 防止删除或禁用最后一个超级管理员。

验收标准：

- 超级管理员可完成用户 CRUD。
- 普通用户不能访问用户管理接口。
- 邮箱唯一性由后端校验。
- 删除用户时对关联数据有明确处理策略。

### 8.3 角色与权限

目标：形成可复用 RBAC 权限体系，替代单一 `is_superuser` 判断。

功能：

- 角色列表、新增、编辑、删除。
- 角色启用/禁用。
- 系统内置角色保护。
- 给角色分配菜单、按钮和接口权限码。
- 用户绑定一个或多个角色。
- 超级管理员默认拥有全部权限。
- 普通用户按角色权限访问菜单和接口。

验收标准：

- 普通用户看不到无权限菜单。
- 普通用户直接请求无权限接口返回 403。
- 修改角色权限后，用户重新登录或刷新权限后生效。
- 系统角色不能被误删或破坏。

### 8.4 菜单管理

目标：菜单、路由和权限码统一维护。

功能：

- 菜单树管理。
- 菜单类型：目录、菜单、按钮。
- 菜单字段：标题、路径、路由名、组件、图标、排序、可见、缓存、状态。
- 按钮字段：权限码、排序、状态。
- 当前用户菜单接口 `/menus/me`。
- 当前用户权限码接口 `/permissions/me`。

验收标准：

- 菜单按树结构展示。
- 禁用菜单不展示且不可访问。
- 按钮权限可用于前端操作按钮显隐。
- 后端权限码是最终访问边界。

### 8.5 部门管理

目标：提供组织结构基础能力。

功能：

- 部门树。
- 新增、编辑、删除部门。
- 部门负责人。
- 部门排序。
- 启用/禁用部门。
- 用户绑定部门。

验收标准：

- 部门支持多级结构。
- 有下级部门或用户时，删除需要阻止或给出明确处理。
- 禁用部门后不影响历史数据展示。

### 8.6 字典管理

目标：把常用枚举配置从代码中解耦出来。

功能：

- 字典类型管理。
- 字典项管理。
- 字典项支持标签、值、颜色、排序、状态、扩展数据。
- 前端可按字典类型读取字典项。

验收标准：

- 字典类型 code 唯一。
- 禁用字典项不出现在业务选择项中。
- 前端表格和表单可复用字典数据。

### 8.7 系统参数

目标：支持常用系统配置在线维护。

功能：

- 参数列表、新增、编辑、删除。
- 参数 key 唯一。
- 参数分组。
- 参数类型：string、number、boolean、json。
- 公开参数可被前端匿名或登录后读取。
- 系统参数有保护规则。

验收标准：

- 非公开参数不通过公开接口返回。
- 系统参数不能被普通删除。
- JSON 类型参数需校验格式。

### 8.8 日志审计

目标：满足生产后台的基础审计要求。

功能：

- 登录日志：账号、用户、IP、UA、状态、失败原因、时间。
- 操作日志：用户、模块、动作、方法、路径、状态码、耗时、IP、UA、请求摘要、响应摘要。
- 日志列表筛选和分页。
- 敏感字段脱敏。
- 日志导出作为 v1.x 增强。

验收标准：

- 登录成功和失败均有记录。
- 关键增删改接口有操作日志。
- 密码、token 等敏感信息不进入日志明文。

### 8.9 文件管理

目标：提供后台常用附件能力。

功能：

- 文件上传。
- 文件列表。
- 文件下载/访问。
- 文件删除。
- 记录文件名、类型、后缀、大小、hash、存储路径、上传人、公开状态。
- 本地存储为 v1.0 默认方案。
- 预留 MinIO/S3 扩展点。

验收标准：

- 上传大小和类型可配置。
- 删除文件同时处理数据库记录和存储文件。
- 私有文件不能被未授权用户访问。

### 8.10 通知公告与站内消息

目标：支持后台发布公告，并向用户形成消息触达。

功能：

- 公告列表、新增、编辑、删除。
- 公告状态：草稿、已发布、已下线。
- 公告类型和优先级。
- 发布公告后生成用户站内消息。
- 用户消息列表、详情、标记已读。
- 未读数展示。

验收标准：

- 草稿不会出现在用户消息中。
- 发布后目标用户能收到消息。
- 用户可标记单条或全部消息为已读。

### 8.11 仪表盘

目标：登录后提供真实系统概览，而不是纯演示数据。

功能：

- 用户数、角色数、文件数、消息数等核心指标。
- 最近登录、最近操作、最近公告。
- 基础趋势图可使用真实聚合数据。

验收标准：

- 仪表盘数据来自真实 API。
- 无数据时展示友好空状态。

### 8.12 示例业务模块 Items

目标：作为开发者新增业务模块的标准参考。

功能：

- 列表、搜索、分页。
- 新增、编辑、删除。
- 权限码控制。
- 导入、导出示例。
- 后端模型、迁移、API、前端页面、API 类型完整闭环。

验收标准：

- 开发者可照着 Items 在 1 小时内新增一个简单 CRUD 模块。
- Items 文档说明每一层应该改哪里。

## 9. 非功能需求

### 9.1 安全

- 所有管理接口默认需要登录。
- 所有管理接口必须进行后端权限校验。
- 密码使用安全 hash 存储。
- Token、密码、验证码等敏感信息不进入日志。
- CORS、环境变量、默认密钥在生产环境有安全检查。

### 9.2 性能

- 常规列表接口默认分页。
- 菜单、权限、字典、参数接口为后续缓存预留结构。
- 单机部署可支撑中小型内部后台的常规访问。

### 9.3 可维护性

- 后端模块按路由、模型、CRUD/服务、测试组织。
- 前端模块按 api、router、views、locales 组织。
- OpenAPI 是前后端类型契约来源。
- 新增模块必须有迁移、权限码、菜单、API、页面和文档。

### 9.4 可部署性

- 本地开发支持前后端分离启动。
- Docker Compose 支持数据库、后端、前端最小部署。
- `.env.example` 覆盖必要配置。
- README 命令与实际项目保持一致。

## 12. 风险与取舍

| 风险 | 说明 | 应对 |
| --- | --- | --- |
| 照搬 Litchi 导致范围失控 | Litchi 是完整快速开发平台，当前项目不应一次追齐 | v1.0 只做基座，平台化能力进入路线图 |
| 权限前后端不一致 | 菜单隐藏不能替代接口鉴权 | 以后端权限码为准，前端只做体验 |
| 初始化数据漂移 | 手工维护菜单和权限容易遗漏 | 初始化脚本、迁移、权限清单文档同步 |
| 日志泄露敏感信息 | 操作日志容易记录密码/token | 统一脱敏策略和测试用例 |
| 文件访问安全 | 本地公开路径可能绕过鉴权 | 区分公开/私有文件，私有文件走授权接口 |
| 功能多但质量松散 | 模块数量增加后测试不足 | v1.0 把测试和文档作为发布门槛 |
