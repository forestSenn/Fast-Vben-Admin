## 2. 核心结论

当前项目已经具备一个全栈后台模板的基本闭环：登录、当前用户、用户管理、Item 示例 CRUD、个人设置、OpenAPI 类型生成、Docker 和测试基础。这对开发者来说已经不是空架子，但如果目标是“用户拿来就能直接用”，功能仍然偏简单。

真正让大众用户反复花时间补的能力不是 Item 示例，而是后台系统的通用基础设施：

- 角色、权限、菜单。
- 部门、岗位、组织结构。
- 数据字典和系统参数。
- 操作日志、登录日志、审计追踪。
- 文件上传、附件管理、对象存储适配。
- 通知公告、站内消息。
- 数据导入、导出。
- 初始化向导、默认数据和生产安全检查。
- 更完整的开发文档、部署文档和模块扩展范式。

因此，本 PRD 将产品目标从“FastAPI + Vben 整合模板”升级为“多数后台项目可以直接二次开发和交付的系统基座”。它不做低代码平台，也不在 v1.0 做复杂多租户和工作流，但要把 80% 中后台都会重复搭建的必要功能内置好。

## 3. 产品定位

Fast Vben Admin 是面向开发者、中小团队、外包交付团队的开箱即用全栈中后台基座。

它提供一套可运行、可登录、可配置权限、可管理组织用户、可审计、可上传文件、可扩展业务模块、可 Docker 部署的完整系统。用户克隆项目后，应能直接把它作为内部管理系统、SaaS 管理端、数据运营后台、业务审批后台的起点，而不是只拿到一个登录页和示例表格。

### 3.1 一句话愿景

让开发者拿到项目当天就能开始写业务，而不是先花一周补账号、权限、菜单、日志、上传、配置和部署这些通用基建。

### 3.2 产品边界

本项目要做：

- 做可直接用于二次开发的后台系统底座。
- 做通用后台基建的标准实现。
- 做清晰的前后端契约、权限模型和扩展文档。
- 做稳定的本地开发和生产部署体验。

本项目不做：

- 不做低代码页面设计器。
- 不做复杂工作流引擎。
- 不做 CRM、ERP、OA 等垂直业务套件。
- v1.0 不做复杂多租户。
- v1.0 不做多数据库官方适配。
- v1.0 不做 OAuth2 / OIDC 第三方登录。

### 4.2 主要缺口

| 缺口 | 为什么重要 |
| --- | --- |
| RBAC 角色权限 | 几乎所有后台都需要，不应让用户从零补 |
| 动态菜单 | 后台菜单通常和权限绑定，是系统管理核心 |
| 部门 / 组织 | 用户管理缺少组织归属，很多业务无法直接落地 |
| 字典管理 | 状态、类型、枚举项需要运营可配置 |
| 参数配置 | 系统名称、开关、默认值、业务参数需要后台可配置 |
| 日志审计 | 生产后台必须知道谁在什么时间做了什么 |
| 文件上传 | 头像、附件、图片、合同等都是高频需求 |
| 通知公告 | 常见内部系统需要公告或站内提醒 |
| 导入导出 | 表格数据导入导出是后台刚需 |
| 初始化向导 | 用户第一次启动需要明确默认账号、安全变量和系统信息 |
| 发布文档 | 要达到开箱即用，README 之外还要有开发、部署、API 契约文档 |

## 5. 目标用户

### 5.1 主要用户

1. 独立开发者
   - 需要快速启动一个可商用改造的后台。
   - 希望减少重复搭建账号、权限、菜单、上传、日志的时间。

2. 中小团队技术负责人
   - 希望团队新项目统一技术栈和工程规范。
   - 需要一个有权限、组织、审计、部署基础的标准基座。

3. 外包和项目交付团队
   - 经常交付管理后台。
   - 关心启动速度、默认功能完整度、页面观感和可维护性。

4. Python / Vue 学习者
   - 希望学习真实全栈项目结构。
   - 需要前后端联动、数据库迁移、权限和部署的完整示例。

### 5.2 非目标用户

- 只需要纯 API 模板的用户。
- 只需要纯前端 Vben 模板的用户。
- 需要大型 ERP / OA / CRM 成品系统的用户。
- 强依赖 React、Django、NestJS、Java Spring Boot 的团队。

## 6. 产品目标

### 6.1 v1.0 目标

v1.0 的目标是达到“拿来就能二次开发”的实用程度，而不是只展示技术栈。

用户应能做到：

1. 克隆项目后按 README 在 10 分钟内启动完整系统。
2. 使用默认管理员登录后台。
3. 创建部门、角色、用户，并分配菜单权限。
4. 使用后台管理系统菜单、字典、参数和公告。
5. 查看登录日志和操作日志。
6. 上传文件，并在业务模块中复用附件能力。
7. 使用示例业务模块作为新增模块参考。
8. 通过 Docker Compose 部署到一台服务器。
9. 按开发文档新增一个完整业务模块。

### 6.2 成功指标

| 指标 | 目标 |
| --- | --- |
| 首次启动时间 | 小于 10 分钟 |
| 默认后台可用性 | 登录后无需改代码即可使用系统管理功能 |
| 通用功能覆盖 | 覆盖 RBAC、组织、字典、配置、日志、文件、通知 |
| 新增业务模块成本 | 开发者能按文档在 1 小时内新增一个简单 CRUD 模块 |
| 测试质量 | 后端核心测试通过，前端构建通过，关键 E2E 通过 |
| 部署体验 | Docker Compose 可完成生产级最小部署 |

## 7. 功能范围

### 7.1 v1.0 必做功能

#### 7.1.1 认证与账号

目标：提供安全、完整、可直接用于后台系统的账号闭环。

功能：

- 邮箱 + 密码登录。
- JWT access token。
- 退出登录。
- 获取当前用户。
- 更新个人资料。
- 修改密码。
- 忘记密码。
- 重置密码。
- 默认超级管理员初始化。
- 禁用用户不可登录。
- 登录失败统一错误提示。

验收标准：

- 默认管理员可以登录。
- 禁用用户无法登录和访问接口。
- 修改密码后旧密码失效。
- 重置密码 token 有过期时间。
- 前端刷新后可恢复登录态。

#### 7.1.2 用户管理

目标：管理员可直接维护系统用户。

功能：

- 用户列表，支持分页、搜索。
- 新增用户。
- 编辑用户基础信息。
- 重置用户密码。
- 启用 / 禁用用户。
- 删除用户。
- 分配部门。
- 分配角色。
- 标识超级管理员。
- 防止删除最后一个超级管理员。

验收标准：

- 超级管理员可完成用户 CRUD。
- 普通用户不能访问用户管理接口。
- 邮箱唯一性由后端校验。
- 删除用户有明确关联数据处理策略。

#### 7.1.3 角色与权限

目标：从 `is_superuser` 升级到通用 RBAC，解决绝大多数后台权限需求。

功能：

- 角色列表。
- 新增、编辑、删除角色。
- 启用 / 禁用角色。
- 给角色分配菜单权限。
- 给角色分配按钮 / 接口权限码。
- 用户绑定一个或多个角色。
- 超级管理员默认拥有全部权限。
- 普通用户按角色权限访问菜单和接口。

内置角色：

- `super_admin`: 超级管理员。
- `admin`: 系统管理员。
- `user`: 普通用户。

验收标准：

- 普通用户看不到无权限菜单。
- 普通用户直接请求无权限接口返回 403。
- 修改角色权限后，用户重新登录或刷新权限后生效。
- 不能删除系统内置关键角色，或删除时有保护规则。

#### 7.1.4 菜单管理

目标：支持后台菜单和权限统一维护。

功能：

- 菜单树管理。
- 菜单类型：目录、菜单、按钮。
- 菜单字段：标题、路由、组件、图标、排序、权限码、是否隐藏、是否缓存、状态。
- 角色分配菜单。
- 前端根据接口返回的菜单生成导航。
- 本地静态路由保留兜底能力。

验收标准：

- 管理员可新增和编辑菜单。
- 菜单排序生效。
- 用户登录后只看到有权限菜单。
- 按钮权限可用于页面操作按钮显隐。

#### 7.1.5 部门与组织

目标：解决用户归属和组织树的通用需求。

功能：

- 部门树。
- 新增、编辑、删除部门。
- 部门启用 / 禁用。
- 部门负责人。
- 用户归属部门。
- 用户列表可按部门筛选。

验收标准：

- 用户可绑定部门。
- 删除部门前校验是否存在子部门或用户。
- 部门树可用于用户管理筛选。

#### 7.1.6 字典管理

目标：让常见枚举项可配置，减少硬编码。

功能：

- 字典类型管理。
- 字典项管理。
- 字典项排序。
- 字典项启用 / 禁用。
- 前端提供字典读取工具。
- 字典缓存和刷新机制。

典型字典：

- 用户状态。
- 性别。
- 是否。
- 业务状态。
- 文件类型。

验收标准：

- 页面下拉选项可以从字典读取。
- 禁用字典项不出现在新建表单中。
- 字典更新后前端可刷新缓存。

#### 7.1.7 系统参数配置

目标：把常见系统开关和基础信息放到后台维护。

功能：

- 参数列表。
- 新增、编辑、删除参数。
- 参数分组。
- 参数类型：字符串、数字、布尔、JSON。
- 系统内置参数不可随意删除。
- 后端提供统一读取配置的服务。

内置参数建议：

- 系统名称。
- Logo 地址。
- 是否开放注册，默认关闭。
- 默认分页大小。
- 文件上传大小限制。
- 密码策略。

验收标准：

- 管理员可修改非敏感参数。
- 敏感配置不通过前端展示明文。
- 系统参数有缓存和刷新机制。

#### 7.1.8 操作日志与登录日志

目标：满足生产后台的基础审计要求。

功能：

- 登录日志：用户、IP、User-Agent、结果、失败原因、时间。
- 操作日志：用户、模块、动作、方法、路径、状态码、耗时、IP、时间。
- 关键操作记录请求摘要。
- 日志列表分页、搜索、筛选。
- 日志详情。
- 日志清理策略。

验收标准：

- 登录成功和失败都有记录。
- 用户新增、删除、角色分配等关键操作有记录。
- 日志不记录密码、token、密钥等敏感信息。
- 普通用户不能查看系统日志。

#### 7.1.9 文件上传与附件管理

目标：提供后台系统最常见的文件能力。

功能：

- 本地文件存储。
- 文件上传 API。
- 文件访问 API。
- 文件列表。
- 文件删除。
- 文件大小和后缀限制。
- 文件哈希和元数据记录。
- 头像上传。
- 预留 S3 / MinIO / OSS 适配层。

验收标准：

- 用户可上传头像。
- 业务模块可复用附件字段。
- 非法文件类型被拒绝。
- 删除文件时数据库和实际存储策略清晰。

#### 7.1.10 通知公告

目标：提供简单可用的站内通知和公告能力。

功能：

- 公告列表。
- 新增、编辑、发布、撤回公告。
- 公告类型和优先级。
- 用户端公告展示。
- 站内消息列表。
- 消息已读 / 未读。

验收标准：

- 管理员可发布公告。
- 登录用户可看到有效公告。
- 用户可标记消息已读。

#### 7.1.11 数据导入导出

目标：覆盖中后台高频表格操作。

功能：

- 用户列表导出。
- 示例业务数据导出。
- CSV / XLSX 导出。
- 用户或示例数据导入模板下载。
- 导入结果反馈，包含成功数、失败数、失败原因。

验收标准：

- 导出文件字段清晰。
- 导入失败不会产生半截不可诊断数据。
- 大数据量导出有分页或异步策略说明。

#### 7.1.12 仪表盘

目标：让后台首页有真实系统价值，而不是纯 Mock 图表。

功能：

- 当前登录用户信息。
- API 健康状态。
- 用户数量。
- 角色数量。
- 文件数量。
- 今日登录次数。
- 今日操作次数。
- 快捷入口。

验收标准：

- 仪表盘数据来自真实 API。
- 无核心 Mock 数据残留。
- 普通用户和管理员看到的内容按权限区分。

#### 7.1.13 示例业务模块

目标：提供一个可复制的业务模块范式。

建议保留并增强 `Items`：

- 列表。
- 新增。
- 编辑。
- 删除。
- 详情。
- 搜索。
- 分页。
- 所属用户。
- 附件。
- 导入导出。
- 权限码控制。
- 操作日志记录。

验收标准：

- 开发者可以按 Items 复制出新的业务模块。
- 后端模型、迁移、CRUD、API、测试、前端页面、权限和菜单完整成套。

## 8. 页面规划

### 8.0 前端实现原则

所有新增页面必须严格遵循 `参考项目/vue-vben-admin` 的样式风格、交互模式和代码写法。Fast Vben Admin 的产品目标是把 Vben 的成熟中后台体验接入真实 FastAPI 后端，而不是重新设计一套后台 UI。

要求：

- 页面布局沿用 Vben Admin 现有主布局、认证布局、菜单、顶部栏、标签页、面包屑、主题和暗色模式。
- 列表页沿用 Vben 风格的查询区、工具栏、表格、分页、操作列和空状态。
- 新增 / 编辑优先使用 Vben 体系中已有的 Modal、Drawer、Form、Table、Descriptions、Button、Icon、Tag、Switch、Dropdown 等组件组合。
- 页面交互沿用参考项目习惯，包括 loading、empty、error、confirm、message、notification 等状态处理。
- 路由、菜单、权限 meta、store、api、views 的组织方式参考 `参考项目/vue-vben-admin`，不另起一套目录和命名风格。
- 样式优先使用项目已有 token、CSS 变量、工具类和组件能力，避免在业务页面写大量孤立样式。
- 图标使用 Vben 现有图标体系或项目已集成图标库，不随意引入新的图标方案。
- 表单校验、请求封装、错误处理、国际化文案应复用已有机制，不在页面组件中散落实现。
- 新页面不得保留参考项目无关 Mock 数据；可以参考其页面结构和组件写法，但数据必须来自真实 API 或明确的本地配置。
- 如果参考项目中已有同类页面，例如用户、角色、菜单、字典、表格 CRUD、详情页，应优先复用其页面结构和交互模式后再接入本项目 API。

验收标准：

- 新增页面视觉上与 Vben Admin 默认后台保持一致。
- 同类操作在不同页面的按钮位置、弹窗形式、表格操作、删除确认和反馈提示一致。
- 页面代码结构能让熟悉 Vben Admin 的开发者快速理解。
- 不出现与 Vben 体系割裂的自定义 UI 风格。

## 9. 权限模型

### 9.1 权限层级

v1.0 使用 RBAC：

- User: 用户。
- Role: 角色。
- Permission: 权限码。
- Menu: 菜单和按钮权限载体。
- Department: 组织归属。

用户通过角色获得权限。菜单和按钮绑定权限码。后端接口通过权限码校验访问权。

### 9.2 权限码规范

```text

system:user:update
```

### 9.3 后端原则

- 后端是权限最终可信来源。
- 前端隐藏菜单和按钮只做体验优化。
- 所有管理接口必须在后端鉴权。
- 超级管理员绕过普通权限校验。
- 系统内置关键数据需要保护。

### 10.2 基础字段

所有业务表建议包含：

- `id`
- `created_at`
- `updated_at`
- `created_by`
- `updated_by`
- `is_deleted` 可选，v1.0 默认不强制软删。

### 10.3 关键约束

- 用户邮箱唯一。
- 角色编码唯一。
- 菜单权限码唯一。
- 字典类型编码唯一。
- 系统参数 key 唯一。
- 部门不允许循环父子关系。
- 删除角色前需校验用户绑定关系。
- 删除菜单前需校验子菜单。
- 日志不可被普通业务操作修改。

## 11. API 契约

### 11.1 通用响应原则

- API 前缀统一为 `/api/v1`。
- 时间字段使用 ISO 8601。
- ID 使用字符串传输。
- 列表接口必须分页。
- 错误响应统一。
- OpenAPI 是前端类型来源。

### 11.2 错误响应

```json
{
  "code": "USER_NOT_FOUND",
  "message": "用户不存在",
  "details": {}
}
```

### 11.3 分页响应

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

### 11.4 常用状态码

| 状态码 | 场景 |
| --- | --- |
| 200 | 查询或更新成功 |
| 201 | 创建成功 |
| 204 | 删除成功 |
| 400 | 业务规则不满足 |
| 401 | 未登录或 token 无效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 唯一性冲突 |
| 422 | 字段校验失败 |

## 12. 非功能需求

### 12.1 安全

- 密码哈希存储。
- 生产环境禁止默认密钥。
- CORS 可配置。
- 后端接口必须鉴权。
- 日志脱敏。
- 文件上传限制大小、类型和路径。
- Token、密码、密钥不得写入日志。
- 删除关键系统数据需要保护规则。

### 12.2 性能

- 列表接口分页。
- 菜单、权限、字典、系统参数可缓存。
- 前端路由按需加载。
- 文件下载走静态服务或对象存储能力。
- 日志表需要索引和清理策略。

### 12.3 可维护性

- 后端模块化组织，避免所有业务堆在单个 `crud.py`。
- 前端页面、API、状态、组件分层清晰。
- 生成代码和手写代码隔离。
- 示例模块必须保持可复制。
- 文档和实现同步维护。

### 12.4 兼容性

- 支持 Windows、macOS、Linux 开发。
- Docker Compose 是首选统一环境。
- 支持现代浏览器，不支持 IE。

## 15. 风险与控制

| 风险 | 表现 | 控制方式 |
| --- | --- | --- |
| 范围膨胀 | 想一次做多租户、工作流、低代码 | v1.0 只做通用后台基建 |
| RBAC 复杂化 | 角色、权限、菜单设计过度 | 采用经典 RBAC，不做数据权限表达式引擎 |
| 前端动态路由不稳定 | 菜单和路由生成出错 | 保留静态兜底路由，动态菜单分阶段上线 |
| 日志影响性能 | 操作日志写入过多 | 只记录关键接口，预留异步写入 |
| 文件存储耦合 | 本地存储后续难迁移 | 设计 storage provider 抽象 |
| 文档滞后 | 用户看文档启动失败 | 每个里程碑同步更新文档 |
| 模板变成成品系统 | 内置过多业务功能 | 保持业务示例最小，重点做系统基建 |
